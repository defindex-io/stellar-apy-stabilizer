# BoostTreasury Contract + Workspace Restructure

**Date:** 2026-04-16
**Phase:** 2 of APY Stabilizer
**Status:** Design approved, pending implementation

---

## Summary

Restructure the `apy-stabilizer` repo from a single-contract project into a Cargo workspace hosting multiple Soroban contracts. Add a new **BoostTreasury** contract that manages marketing-campaign funds used to boost DeFindex vault APY when strategy yields fall below the partner's target.

This enables Phase 2 of the APY stabilizer system: when a vault's strategy APY drops below its target, fees alone can't reach the target. The boost mechanism transfers USDC (or the vault's underlying asset) directly to the vault's idle balance, increasing PPS for all shareholders without minting new shares.

---

## Context

### Phase 1 (Complete)

- **FeeProxy contract** (Soroban) — single proxy managing all partner vaults, adjusts fees to target APY
- **vault-ops module** (NestJS, defindex-api) — cron-triggered fee adjustment service
- **Deployment coordination:** Contract deploys AFTER Phase 2 so the indexer pipeline can be set up in parallel

### Phase 2 (This Spec)

- **Cargo workspace restructure** — prepare for multiple contracts
- **BoostTreasury contract** — fund custody + boost execution
- **boost.service.ts + POST /vault-ops/boost endpoint** (separate spec, after contract approval)

### Key Design Decisions Already Made

- **Single-asset vaults ONLY** — no multi-asset support (none exist currently)
- **No `boost()` method on FeeProxy** — boost just transfers tokens to the vault; idle funds naturally increase PPS; no rebalance needed
- **No withdraw() for depositors** — admin uses `transfer()` for any refunds; avoids per-depositor tracking complexity
- **Idle funds DON'T earn yield in Phase 2** — DeFindex-vault-as-treasury optimization deferred to a future version
- **Same `FEE_MANAGER_SECRET_KEY`** signs both FeeProxy and BoostTreasury transactions
- **No campaign expiry dates** — manual deactivation via `update_campaign(active: false)`
- **No minimum boost amount**
- **`boost()` is manager-only** — admin can use `transfer()` for emergency manual boosts

### Design Rationale

**Fully independent contracts** (no shared crate). Each contract has its own `storage.rs` with its own TTL constants and helpers. The ~10 lines of TTL boilerplate that would be shared aren't worth the coupling complexity. Each contract can be audited independently and can diverge if needed.

## BoostTreasury Contract

### Purpose

Hold marketing campaign funds per vault. Allow the bot to transfer funds to vaults as APY boosts. Allow admin to withdraw funds for refunds or reallocation.

### Roles

| Role | Address | Purpose |
|------|---------|---------|
| **Admin** | Multisig wallet | Full control: register/update/unregister campaigns, transfer funds out, set manager |
| **Manager** | `FEE_MANAGER_SECRET_KEY` wallet (same as FeeProxy's fee_manager) | Can only call `boost()` |

### Storage

```rust
// storage.rs

const DAY_IN_LEDGERS: u32 = 17_280;
const INSTANCE_BUMP_AMOUNT: u32 = 30 * DAY_IN_LEDGERS;
const INSTANCE_LIFETIME_THRESHOLD: u32 = INSTANCE_BUMP_AMOUNT - DAY_IN_LEDGERS;
const PERSISTENT_BUMP_AMOUNT: u32 = 120 * DAY_IN_LEDGERS;
const PERSISTENT_LIFETIME_THRESHOLD: u32 = PERSISTENT_BUMP_AMOUNT - 20 * DAY_IN_LEDGERS;

#[contracttype]
#[derive(Clone)]
pub enum DataKey {
    Admin,
    Manager,
    Campaign(Address),              // keyed by vault address
}

#[contracttype]
#[derive(Clone, Debug, PartialEq, Eq)]
pub struct Campaign {
    pub active: bool,
    pub asset: Address,             // single asset the vault uses
    pub total_deposited: i128,
    pub total_boosted: i128,
    pub total_withdrawn: i128,
}

impl Campaign {
    pub fn available(&self) -> i128 {
        self.total_deposited - self.total_boosted - self.total_withdrawn
    }
}

// Minimal types mirroring the vault's return from get_assets()
// Only what we need to read — avoids bloating WASM with contractimport!
#[contracttype]
#[derive(Clone)]
pub struct VaultStrategy {
    pub address: Address,
    pub name: String,
    pub paused: bool,
}

#[contracttype]
#[derive(Clone)]
pub struct VaultAssetStrategySet {
    pub address: Address,
    pub strategies: Vec<VaultStrategy>,
}
```

**Storage tiers:**
- **Instance:** `Admin`, `Manager` (global config)
- **Persistent:** `Campaign(vault)` (per-campaign, independent TTL)

### Contract Methods

```rust
#[contract]
pub struct BoostTreasury;

#[contractimpl]
impl BoostTreasury {
    // --- Constructor ---

    pub fn __constructor(env: Env, admin: Address, manager: Address) {
        admin.require_auth();
        storage::set_admin(&env, &admin);
        storage::set_manager(&env, &manager);
    }

    // --- Read-only ---

    pub fn get_admin(env: Env) -> Address;
    pub fn get_manager(env: Env) -> Address;
    pub fn get_campaign(env: Env, vault: Address) -> Campaign;
        // Panics with CampaignNotRegistered if not found

    // --- Admin-only ---

    pub fn set_manager(env: Env, new_manager: Address);
        // admin.require_auth()
        // Emits ManagerUpdated

    pub fn register_campaign(env: Env, vault: Address);
        // admin.require_auth()
        // Panics if campaign already exists
        // Calls vault.get_assets() — expects Vec<VaultAssetStrategySet>
        // Rejects if len() != 1 (multi-asset not supported)
        // Stores Campaign { active: true, asset: <vault's asset>, totals: 0 }
        // Emits CampaignRegistered

    pub fn update_campaign(env: Env, vault: Address, active: bool);
        // admin.require_auth()
        // Panics if campaign not registered
        // Updates active status
        // Emits CampaignUpdated

    pub fn unregister_campaign(env: Env, vault: Address);
        // admin.require_auth()
        // Panics if available() != 0 (admin must transfer() remaining funds first)
        // Removes from storage
        // Emits CampaignUnregistered

    pub fn transfer(env: Env, vault: Address, amount: i128, to: Address);
        // admin.require_auth()
        // Panics if amount <= 0 (InvalidAmount)
        // Panics if amount > available() (InsufficientBudget)
        // token.transfer(contract_address, to, amount)
        // total_withdrawn += amount
        // Emits Transferred

    // --- Anyone (authenticated) ---

    pub fn deposit(env: Env, caller: Address, vault: Address, amount: i128);
        // caller.require_auth()
        // Panics if campaign not registered or not active
        // Panics if amount <= 0
        // token.transfer(caller, contract_address, amount) — uses campaign.asset
        // total_deposited += amount
        // Emits Deposited

    // --- Manager-only ---

    pub fn boost(env: Env, vault: Address, amount: i128);
        // let manager = get_manager(env); manager.require_auth()
        // Panics if campaign not registered or not active
        // Panics if amount <= 0
        // Panics if amount > available()
        // token.transfer(contract_address, vault, amount) — uses campaign.asset
        // total_boosted += amount
        // Emits Boosted
}
```

### Auth Helper

```rust
// Private helper in lib.rs

fn require_manager(env: &Env) {
    let manager = storage::get_manager(env);
    manager.require_auth();
}

fn require_admin(env: &Env) {
    let admin = storage::get_admin(env);
    admin.require_auth();
}
```

Simpler than FeeProxy — no per-vault admin, just global roles.

### Token Transfer Pattern

```rust
use soroban_sdk::token;

// In deposit()
token::Client::new(&env, &campaign.asset)
    .transfer(&caller, &env.current_contract_address(), &amount);

// In boost() and transfer()
token::Client::new(&env, &campaign.asset)
    .transfer(&env.current_contract_address(), &to, &amount);
```

### TTL Extension

Every write method calls `extend_instance_ttl(&env)` (matching FeeProxy pattern). Per-campaign TTL is extended automatically on `get_campaign` / `set_campaign` operations in the storage layer.

**Coverage:**

The bot-side implementation will be specced separately after contract approval. Preview for context:

### New Services in `vault-ops/` Module

```
src/vault-ops/
├── boost.service.ts                # Phase 2 — boost cycle orchestration
├── boost-treasury.service.ts       # Phase 2 — BoostTreasury contract wrapper
```

### New Endpoint

```
POST /vault-ops/boost
  Auth: CRON, ADMIN
  Triggered: every 1h via separate Google Cloud Scheduler cron
```

### Flow per cycle (every hour)

1. Discover managed vaults (same query as fee stabilizer)
2. For each vault, check if `strategy_apy_7d < target_apy` (boost needed)
3. Check BoostTreasury for active campaign + available budget
4. Calculate per-cycle boost: `(tvl × apy_gap) / cycles_per_year` (8760 if hourly)
5. Clamp to available budget
6. Call `BoostTreasury.boost(vault, amount)` signed by MANAGER key
7. Return per-vault results

### Infrastructure

- Same `FEE_MANAGER_SECRET_KEY` as FeeProxy
- New env var: `BOOST_TREASURY_ADDRESS` (per network)
- Separate cron schedule from fee stabilizer (1h vs 6-12h)

## Security Considerations

### Trust Model

- **Admin (multisig):** Full control over campaigns, can transfer funds out. Highest trust.
- **Manager (bot wallet):** Can only call `boost()`. Bounded blast radius — if compromised, attacker can drain campaigns to registered vaults only. Funds end up in vaults (not attacker-controlled). Partners can reclaim via vault operations.

### Attack Vectors

**Manager key compromise:**
- Attacker can call `boost()` to drain campaigns to registered vaults
- Funds go to legitimate vault addresses — not stolen, just accelerated distribution
- Response: admin calls `set_manager()` to rotate key, calls `update_campaign(active: false)` to pause

**Admin key compromise:**
- Attacker can transfer all funds to any address via `transfer()`
- Mitigation: admin is a multisig wallet with multiple signers

**Malicious vault registered:**
- Admin controls registration; only registered vaults can receive boosts
- `register_campaign` validates the vault is a real single-asset vault via `get_assets()`
- Risk limited to admin diligence — same as any treasury admin operation

**Zero/negative amount attacks:**
- All amount parameters validated: `amount > 0` required
- Integer overflow prevented by Soroban's `overflow-checks = true` in release profile

### TTL Management

- Contract is not upgradable (same as FeeProxy)
- Every write method extends instance TTL
- Per-campaign TTL extended on every read/write
- If the contract is idle for > 30 days, instance storage archives (can be restored)
- Recommendation: admin or manager should interact with the contract at least monthly to keep it warm
